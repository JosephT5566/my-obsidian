	```javascript
replicas: 3

	```
	- 有 3 個 Pod 在跑
	- 都用這個 image
	- 壞了會補

	- 建立 Pod
	- health check
	- kill / restart
	- reschedule
3. Rollout 機制（Rolling Update）
	- 逐步起新 Pod
	- 等 readiness
	- 再殺舊 Pod

	- Service
	- Endpoint
	- kube-proxy
5. 排程（schedule 到哪個 node / pool）
	- resource request / limit
	- node selector / affinity
	- taint / toleration
# Pod
> Pod： 是一個「豆莢」，裡面裝著一個或多個「豆子」（容器, container）。這些豆子緊密相連，共享一個外殼（Pod 的資源）。
1. 組成
	1. 一個 Pod 可以包含一個或多個容器。
	2. 這些容器會共享相同的網路（IP 位址、埠號）和儲存資源。
	3. 通常，如果多個容器需要緊密協作，而且它們的生命週期是綁定在一起的，那麼它們就會被放在同一個 Pod 裡
2. 特點
	1. **一台正在跑程式的 VM（但更輕）**
		1. **Image + runtime environment + resource + lifecycle**
	2. 有生命週期（生 / 死）
	3. 佔用 CPU / memory
	4. 會壞、會被重建
3. 生命週期管理： Kubernetes 不會直接管理**單一的容器**，而是透過 Pod 來管理容器的生命週期。當你說要啟動一個應用程式時，實際上是在啟動一個 Pod。**Pod 內的 containers 都跟 Pod 共生死**
4. 通訊與協作：在同一個 Pod 內的容器可以透過 `localhost` 互相通訊，就像在同一台機器上。

### Container to container
Pod 內的 containers 是透過 `localhost` 互動的，一個 Pod = 一個 network namespace
	1. 共用 **同一個 IP**
	2. 共用 **同一個 loopback（127.0.0.1）**
### **pod-to-pod (same node)**
> *All pods can communicate with all other pods, whether they are on the same *[*node*](https://kubernetes.io/docs/concepts/architecture/nodes/)* or on different nodes. Pods can communicate with each other directly, without the use of proxies or address translation (NAT).*

### **CNI（Container Network Interface）**
- 在 node 上設定 routing / overlay network
- 確保 Pod IP 可被其他 node reach
常見 CNI：
- Calico
- Cilium
- Flannel
- Weave

# Node & pool
[Kubernetes 架構圖解：Cluster、Node Pool、Node、Pod 一次搞懂](https://vocus.cc/article/688b1936fd897800010f84fe)
## Node
Node (節點) 是構成整個 cluster 的基本工作單元。
一個 Node 可以是一台實體伺服器，也可以是一個虛擬機器 (VM)。它負責提供運算資源 (CPU, 記憶體) 和儲存空間，來運行你的應用程式。
可以把 Node 視為 K8s cluster 的骨架，所有的應用程式 (Pods) 最終都是運行在這些 Node 上。
每個 Node 都是一台電腦（通常是虛擬機，如 Google Cloud VM），能夠執行多個容器。
Node 的 4 個基本軟體元件：
- pod
- kubelet：跟 Kubernetes 控制平面溝通
- 容器運行環境（如 Docker 或 containerd）：負責執行容器
- kube-proxy：處理網路與服務轉送
## Pool
一組具有相同配置的「節點 (Nodes)」的集合，簡單說就是「有相同規格的工作機群組」。
> 可以把一個 K8s cluster 想像成一家大型公司，而一個 Node Pool 就是公司內部的一個**「特定職能小組」或「團隊」**。這個團隊裡的所有成員都擁有相同的技能、工具和設備（例如：相同的 CPU、記憶體、硬碟大小），因此他們適合處理同一類型的工作。
Cluster 包含一個或多個「Node Pool」。每個 Node Pool 裡面有多台節點（本質上是 VM 虛擬機），用來執行你的容器。<br>可以根據需求建立多個 Node Pool，並為它們設定不同的：
1. 機器型號（CPU、RAM）
2. 作業系統（Linux、Windows）
3. 自動擴展策略
4. 區域（zone）或地區（region）
```javascript
Cluster
├─ Control Plane
│    └─ Scheduler / Controller
│
└─ Node Pools   ← pool 在這裡
		├─ main pool
		│    ├─ Node A
		│    ├─ Node B
		│    └─ Node C
		│         └─ Pod
		│
		├─ batch pool
		└─ gpu pool
```
### Why do we need Node Pool?
在一個大型的 K8s cluster 裡，不會希望所有節點都長得一模一樣。Node Pool 的主要目的就是為了實現資源的異質性 (Heterogeneity) 和管理上的便利性。
1. 處理不同類型的工作負載
	1. 你的應用程式可能有多種不同類型。例如，一個機器學習應用可能需要配備 GPU 的高運算節點，而一個簡單的網站伺服器只需要普通的 CPU 節點。
	2. Node Pool 允許你在同一個叢集裡，建立多個不同配置的節點池。你可以有一個「GPU 節點池」專門跑機器學習任務，另一個「通用節點池」則用來跑你的網站服務。
2. 簡化管理與維護
	1. Node Pool 讓你可以將維護任務應用到整個節點群組，而不是一個一個節點去處理。<br>例如，當你需要升級作業系統或 Kubernetes 版本時，你可以一次性地升級整個節點池。<br>當你想要擴展你的服務時，你只需要增加節點池的數量，而不需要為每個新節點都單獨配置。
3. 優化成本
	1. Node Pool 讓你能夠為不同需求的工作負載，選擇最經濟的機器類型。
	2. 你可以使用較便宜的機器來運行不太重要的開發環境，而使用效能更強、成本更高的機器來運行關鍵的生產服務。
4. 獨立的自動擴展
	1. 你可以為每個 Node Pool 設定獨立的自動擴展規則。<br>例如，當你的網站服務流量增加時，只有「網站節點池」會自動增加節點，而「機器學習節點池」則不受影響。<br>
# Clusters
**單一 Kubernetes 本身「只會管理一個 cluster」**
但在實務上，**公司會同時擁有很多個 K8s clusters**，再用：
- 人的流程（DevOps / SRE）
- 或額外工具（ArgoCD、Fleet、Rancher、Anthos、EKS multi-account）
來「**管理多個 cluster**」
**每個 cluster 裡面通常都會有：**
- Ingress / Load Balancer
- API Gateway / GraphQL Gateway
- Backend microservices
- Workers / CronJobs
- Redis / Kafka /（有時 DB）
典型 prod cluster 內部會是：
1. \[Cloud LB / ELB\]
2. \[Ingress / API Gateway\]
3. \[GraphQL Gateway / BFF\]
4. \[Backend microservices\]
5. \[DB / Cache / MQ（或外部）\]
## 分 clusters 的好處
1. Blast Radius（爆炸半徑）控制
2. 不同 cluster = 不同：
	- IAM role
	- network policy
	- secret
	- account / VPC
3. 資源型 workload 不互相影響
	1. 例如：<span color="orange_bg">ML team 跑一個 training，整個 cluster 的 CPU / memory 都被吃光</span>
4. 升級與實驗自由度
	1. Kubernetes 版本升級
	2. CNI / Ingress controller 換版本
	3. 新架構實驗
5. 組織與責任邊界清楚
## cluster 之間會「互相溝通」嗎？
會，但通常是「有限、刻意設計過的」
Cluster 之間的互動，設計上要「像在呼叫外部系統」
1. 透過 **Public / Internal API**
	> Cluster A → HTTPS → Cluster B
	- payment API
	- user profile API
	- analytics ingestion
2. Message Queue / Event Bus（解耦）
	> Cluster A → Kafka / PubSub → Cluster B
	- 訂單事件
	- 使用者行為
	- 非同步流程
3. 直接 cross-cluster service mesh（少見）

# Service
[Service](https://kubernetes.io/docs/concepts/services-networking/service/)
[\[Kubernetes\] Service Overview](https://godleon.github.io/blog/Kubernetes/k8s-Service-Overview/)
> Expose an application running in your cluster behind a single outward-facing endpoint, even when the workload is split across multiple backends.
## 前言
Service 想解決的問題：
當 workload 變成 micro service(pod) 進到 k8s 後，在存取上會遇到什麼問題？
## **Pod 存取問題**
pod 的生命周期不是並永久的，一旦掛掉之後就不會再自己恢復，因此若使用者針對 pod 有多個 replica 的需求，k8s 中額外設計了像是 `ReplicaSet` or `Deployment` 這一類的 resource object 來確保 pod 掛掉之後可以再重啟。
但 pod 重啟以後會發生什麼問題呢？答案就是 **「IP 也會跟著改變」**
由於 pod IP 可能隨時會變動，因此若是依照傳統 application 的設定方式，使用 ip 設定 DB or web 的位址，上到 k8s 之後可能就很快就無法正常運作。
## How Service resolve this problem?
K8s 在 pod 的前方提供了一個 Service 抽象層，讓外部的服務可以用 **domain name** 的方式存取 pod，而 **domain name ↔ Pod** 這一段問題，就由 Service 來處理。
利用 domain name 來存取終究還是需要一個 IP address，而每個 Service 都會自帶 VIP (virtual IP)，讓 network traffic 有辦法正常送過來，並導到後方真正提供服務的 pod。
當 network traffic 進到 service 後，則是透過 `Label Selector` 決定導到哪些 pods。
## Service types
- `ClusterIP`：提供 cluster 內部存取
- `NodePort`：供外部存取
- `LoadBalancer`：供**外部**存取 (但只限於支援此 type 的 public cloud)
	- Houzz 使用 aws 故有支援，使用這個 type 通常目的是 cross-cluster traffic，如果都在 saas cluster 中的 pods 互相溝通，那就不需要
- `ExternalName`：單純回應給 client 外部的 DNS CName record (並非由 k8s cluster 內部的 pod 服務)
## Create a service
[Kubernetes 那些事 — Service 篇](https://medium.com/andy-blog/kubernetes-%E9%82%A3%E4%BA%9B%E4%BA%8B-service-%E7%AF%87-d19d4c6e945f)
```yaml
apiVersion: v1
kind: Service
metadata:
  name: helloworld
  labels:
    app: frontend
    environment: develop
spec:
  type: NodePort
  ports:
    - protocol: TCP
      port: 8080
      targetPort: 8080
      nodePort: 32700
  selector:
    app: frontend
```
基本上寫法就跟 Pod 差不多，唯一不同的地方就在 spec 的細部設定
- type: 用來決定這個 Service 要以什麼形式建立， type 的值就是上面提到的 4 種
- ports: 用來決定該 Service 要連接哪些 port ，這邊有一些細部的設定
	- protocol: 用來決定連線的網路協議，預設值為 TCP ，當然也可以使用 UDP。
	- port: 建立好的 Service 要以哪個 port 連接到 Pod 上。
	- targetPort: 指的是目標 Pod 的 port ，通常 port 跟 targetPort 會設定一樣。
	- nodePort: 指的是機器上的 Port 要對應到該 Service 上，這個設定要 nodePort 形式的 Service 才有效果，假如今天沒有設定 nodePort 的話，K8s 就會自動開一個機器上的 port 去對應到該 Service ，這邊有個小提醒，nodePort 的範圍在 30000–32767 之間，不在這範圍內的 port 都是不允許的。
- selectors: Service 想要連接到正確的 Pod 就必須要利用 selectors 了，這邊只要原封不動的把 Pod 的 Label 複製上去即可。
	- 一個 Service 只會有「**一組** selector 條件」，但這組 selector 可以匹配「多個 Pods」。
	- selector 可能會含有多個 labels，同時滿足條件才會被選中，如：
		- app: image-classification
		- env: production
		- version: v1
		- tier: backend
	- 多個 pods 被選中的話，就會透過 kube-proxy 做 load balance
## In cluster DNS
[\[Kubernetes\] DNS for Service & Pod](https://godleon.github.io/blog/Kubernetes/k8s-DNS-for-Service-and-Pod/)
> DNS service 在 k8s cluster 內部負責 service discovery 的重責大任，讓使用者可以跳脫 IP based 的思維，以 domain name 的方式來進行 application 的架構設計。
	而 k8s DNS service 其實也是由一個 DNS pod + service 所組成，並且告訴 kubelet 將每個 container 的 DNS resolution 都指向這個 DNS service，藉由此方式，每個 pod 在存取其他 service 時，都會向一開始就安裝好的 DNS pod 進行 domain name 的詢問。
### Domain name
k8s 中有 namespace 的概念，由於不同的 namespace 中可以有同樣名稱的 service or pod，因此 DNS 解析的部份就需要考慮 namespace
k8s cluster domain name，若是未設定，預設就會是 `cluster.local`
如在 GQL service 中設定的 config-override 的 default host `threed-image-classification-service.backend.svc.cluster.local:9090`

最後 Benjamin 猜測可能是 protocol 的問題，目前 service 是設定：
