# 核心概念
OAuth 是一種「授權框架」，允許使用者透過第三方（例如 Google、GitHub）登入應用程式，
而不用把密碼交給應用程式本身。

## ID Token
### 使用方式
當頁面**載入**時呼叫
以 React 來說，可能包在 useEffect 中，確認 `window !== “undefined”`
```javascript
const gis = window.google.accounts.id
gis.initialize({
  clientId: <your client id>,
  callback: (resp) => {
    const idToken = res?.credential;
  }
})
```
使用這系列方法（通常為 **Google One Tap** 或 **Sign In With Google** 按鈕）時，Google 回傳的是一個經過簽名的 **JWT (JSON Web Token)**。
### Q&A
**Q: 用 ID Token 發送給 API 作為驗證手段合適嗎？**
看 **API 目的地** 是哪裡：
- **發送給「你自己的」後端 API：非常合適（甚至是標準做法）。**<br>你的後端收到 ID Token 後，解碼並驗證簽章，就能確認「這個請求確實來自使用者 A」。這是目前實作「登入系統」的主流方式。如 Google cloud function (GCF)
- **發送給「Google 的」 API（如 Google Drive, Calendar）：不合適且通常無效。**<br>如果你要叫 Google 幫你做事，Google API 只認 **Access Token**。
**Q: ****`window.google.accounts.id`**** 是否已淘汰？**
**沒有。** 事實上，它是 Google 目前**最推薦**的身分驗證方式。
Q: **有辦法實現 Implicit Mode 嗎？**
**沒辦法。**
`google.accounts.id` 的設計初衷是 **Authentication (驗證)**，它不會讓你請求細節的 Scopes（例如 `https://www.googleapis.com/auth/calendar`）。
如果你要在 React 中實現 **Implicit Mode**（即在前端取得 Access Token），你必須**同時**或**改用** `google.accounts.oauth2` 模組：
在 React 專案中，通常會這樣混合使用：
1. **登入時**：用 `google.accounts.id` 顯示「Google 登入」按鈕，取得 **ID Token** 傳給後端建立 session（身分證）。
2. **需要操作 Google API 時**：當使用者點擊「同步日曆」按鈕，呼叫 `google.accounts.oauth2.initTokenClient` 彈出授權視窗，取得 **Access Token**（鑰匙）。
## Access Token
如果是**純靜態網頁**，且**沒有自己的後端伺服器**，其實可能根本不需要 `google.accounts.id` (ID Token)。直接使用 `google.accounts.oauth2` (Access Token) 即可同時滿足「識別使用者」與「操作 API」的需求。

如果你自己用 Google 官方 OAuth 端點：
```typescript
https://accounts.google.com/o/oauth2/v2/auth
```
### 初始化
同樣是當頁面**載入**時呼叫，
[https://developers.google.com/identity/oauth2/web/reference/js-reference?hl=zh-tw#google.accounts.oauth2.initTokenClient](https://developers.google.com/identity/oauth2/web/reference/js-reference?hl=zh-tw#google.accounts.oauth2.initTokenClient)
```javascript
const gisClient = window.google.accounts.oauth2
gisClient.initTokenClient({
  client_id: <your client id>
  scope: <以空格分隔的範圍清單，可識別應用程式可代表使用者存取的資源。>
  callback: (tokenResponse) => {
    const accessToken = tokenResponse.access_token;
  }
})
```
可以繼續用 access token 取得 user info：
```javascript
callback: async (tokenResponse) => {
  const accessToken = tokenResponse.access_token;
  const res = await fetch("https://www.googleapis.com/oauth2/v3/userinfo", {
        headers: {
            Authorization: `Bearer ${accessToken}`,
        },
    });
  const userInfo = await res.json();
}
```
### 取得 Access Token
當 Token client 初始化完成後，就可以使用 **`requestAccessToken`**** **方法取得 access token
[https://developers.google.com/identity/oauth2/web/reference/js-reference?hl=zh-tw#TokenClient](https://developers.google.com/identity/oauth2/web/reference/js-reference?hl=zh-tw#TokenClient)
```javascript
gisClient.requestAccessToken({ prompt: "consent" });
```
token 約 1 小時後過期。
### 靜默授權 (Silent re-auth)
當 `accessToken` 過期
- **後果**：無法呼叫 Google API，但使用者**依然是登入狀態**。
- **做法**：在呼叫 Google API（如 Calendar）遇到 401 時，不要登出使用者，而是呼叫 `tokenClient.requestAccessToken({ prompt: 'none' })`。
	- `prompt: 'none'`：會嘗試靜默獲取新 Token（如果使用者在瀏覽器中還有 Google Session）。
	- 如果靜默獲取失敗，再改用 `prompt: 'consent'` 讓使用者點一下授權。

## Implicit Flow
[選擇使用者授權模型  \|  Authorization  \|  Google for Developers](https://developers.google.com/identity/oauth2/web/guides/choose-authorization-model?hl=zh-tw#choosing_an_authorization_flow)
[https://ithelp.ithome.com.tw/m/articles/10291817](https://ithelp.ithome.com.tw/m/articles/10291817)
核心特性是：**授權伺服器直接在瀏覽器的重導向中，將 Access Token 回傳給前端，而不需要經過後端的「授權碼交換」過程。**
使用 `google.accounts.oauth2.initTokenClient` 其實就是在使用 **Google 版的隱式模式**。
### 運作流程
假設你在 React 網頁中點擊「讀取 Google 日曆」：
1. **使用者發起請求**：你的網頁將使用者引導至 Google 的授權頁面，網址參數包含 `response_type=token`。
2. **使用者授權**：使用者在 Google 頁面點擊「同意」。
3. **直接回傳 Token**：Google 驗證後，直接將瀏覽器重導向回你的網頁，並在網址的 **Fragment (#)** 部分帶上 Access Token（例如：`https://yourapp.com/#access_token=ya29.xxxx`）。
4. **前端讀取**：你的 JavaScript 腳本從網址中抓取這個 Token，從此可以拿它去呼叫 API。
### 為什麼叫「隱式」(Implicit)？
在標準的「授權碼模式」(Authorization Code Mode) 中，前端拿到的是一個 `code`，必須由後端伺服器拿 `code` + `Client Secret` 去換 Token。
但在 **Implicit Mode** 中：
- **省略了交換過程**：不需要 `code`。
- **省略了 Client Secret**：因為前端代碼是公開的，沒辦法藏秘密，所以乾脆不使用 Secret。
- **隱含的意思**：只要你能證明你是這個 Client ID 的合法重導向網址，Google 就「隱含地」相信你並直接發放 Token。
### 相關討論
Google 的新版 JavaScript SDK (GSI) 實際上將其稱為 **「Token Model」**。
- [Google Identity - Use the Token Model](https://developers.google.com/identity/oauth2/web/guides/use-token-model?authuser=1)
- Google 建議如果只是單純的前端 API 調用，使用此模式。但如果你需要長效存取，請改用「Code Model」（對應 Authorization Code Flow）。
- 現在建議 SPA 應改用 **Authorization Code Flow + PKCE**（Proof Key for Code Exchange）。防止 Token 在網址傳遞過程中被竊取。
# Supabase OAuth（封裝後端機制）
當你在前端使用：
```typescript
supabase.auth.signInWithOAuth({ provider: 'google' });
```
Supabase 會在後端代你：
1. 與 Google 完成 OAuth flow。
2. 收到 Google 的 access token / refresh token。
3. 換發 **Supabase 自家的 access token (JWT)** + refresh token。
4. 儲存在瀏覽器並由 SDK 管理。

# Refresh Token 過期會發生什麼？
- Supabase 會拒絕換新 access token。
- SDK 會自動：
	- 清除 session（access/refresh token）
	- 觸發 `SIGNED_OUT` 狀態事件
	- 使用者被登出。
👉 也就是「一個月自動登出一次」的行為。

## OAuth Proxy
引入中間平台，如 **Cloudflare**，讓驗證架構從「前端直接對 Google」轉變為「前端對中間層，中間層對 Google」。
### 驗證關係的轉變
在這種模式下，中間層（如 Cloudflare Workers）充當了「託管人」。
- **Client Secret**：完全存在中間層的環境變數中，前端完全接觸不到。
- **Refresh Token**：也應該由中間層保管（存放在中間層的數據庫如 KV，或通過 **HttpOnly Cookie** 傳給前端）。
- **Access Token**：中間層向 Google 交換後，可以直接用來代表前端去請求數據，或者發還給前端（但建議同樣放在記憶體中或 Secure Cookie）。
### **前端如何判斷 Signed In？**
方法 A：檢查伺服器 Session 接口 <span color="red">(推薦)</span>
前端在啟動時呼叫中間層的一個 `/api/me` 或 `/api/status` 接口。
- **流程**：前端請求 `/api/status`。
- **中間層判斷**：檢查請求帶來的 Cookie 是否有效。
	- 如果有效：回傳用戶資料（如姓名、頭像）。
	- 如果無效：回傳 `401 Unauthorized`。
- **前端動作**：若收到 `401`，則顯示「登入」按鈕；若收到用戶資料，則進入「已登入」狀態。
方法 B：使用 HttpOnly Cookie
這是最安全的方式。中間層在完成 Token 交換後，將 Session ID 寫入一個 `HttpOnly` 且 `Secure` 的 Cookie。
- **特性**：JavaScript 讀不到這個 Cookie（防止 XSS 攻擊），但每次 `fetch` 請求時瀏覽器會自動帶上它。
- **判斷**：前端不需要「讀取」Token。前端只需要發送一個測試請求，根據回傳碼（200 或 401）來判斷狀態。
### 前端與 proxy 的互動
前端不需要主動帶入 User Email 或 Secret 給 proxy。
當呼叫 Proxy 時，瀏覽器會自動完成大部分工作：
- **Credentials (Credentials: 'include')**：這是最重要的。在 `fetch` 時必須設定，這樣瀏覽器才會把 **HttpOnly Cookie (Session ID)** 自動帶給 Cloudflare Worker。
- **指令與數據**：你只需要告訴 Proxy 你想做什麼（例如：請求哪個 Google 終端點、Body 放什麼）。

# 範例：Supabase 前端取得 Session 與自動 refresh
```typescript
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(SUPABASE_URL, SUPABASE_KEY);

// 初始化時自動載入現有 session
const { data: { session } } = await supabase.auth.getSession();
console.log('Current user:', session?.user?.email);

// 偵測登入狀態改變
supabase.auth.onAuthStateChange((event, session) => {
  if (event === 'SIGNED_OUT') {
    console.log('User signed out');
  } else if (event === 'SIGNED_IN') {
    console.log('User signed in:', session.user.email);
  }
});


```

# 總結重點整理
- OAuth 的「access token」是短期憑證，「refresh token」是長期憑證。
- Supabase 幫你把 Google OAuth 封裝成自己的 token 系統，讓前端也能安全 refresh。
- 若自己實作 OAuth（無後端），不應保存 refresh token，應改用 PKCE + silent reauth。
- Supabase Free Plan 預設 session 約 30 天有效；refresh token 過期後會自動登出。
- 可以自行在前端加上 idle timeout、manual reauth 提升安全性。
