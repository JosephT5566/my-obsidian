# **900+ hours of Learning System Design in 9 Minutes**
[900+ hours of Learning System Design in 9 Minutes](https://www.youtube.com/watch?v=3Pusamd6BO4)
## 1. 無狀態（Statelessness）
無狀態伺服器不會記住先前請求的資訊。每次請求都必須攜帶完成處理所需的資料，通常包含儲存在客戶端的身分驗證 Token。
重點：
- 無狀態是水平擴展（Horizontal Scaling）的重要基礎。
- 如果 Session 資料存在特定伺服器上，使用者之後的請求就必須回到同一台伺服器，這稱為「黏性工作階段」（**Sticky Session**）。
- Sticky sessions limit the load balancer and create a failure risk: if that server dies, its session data may disappear.
- 狀態應該移到 Redis 或其他所有伺服器都能存取的分散式儲存系統 (live in a shared system)。
- 當所有伺服器都可以互相替代時，負載平衡器便能自由分配流量，也可以安全地增加或移除伺服器實例。
面試時應該思考：**每台伺服器保存了哪些狀態？這些狀態真正應該存放在哪裡？**
## 2. 快取（Caching）
All caches make essentially the same **trade-off**: improved speed in exchange for potentially stale data.
重點：
- 快取會把資料副本放在速度更快，或距離使用者更近的位置。
- 瀏覽器快取可以減少客戶端重複下載相同資源。
- CDN 會從距離使用者較近的節點提供不常變動的靜態內容。
- Redis 等應用層快取可以減少大量重複、以讀取為主的資料庫查詢。
- 資料庫內部也可能具備查詢快取。
- 與其死背各種快取技術，不如先找出系統瓶頸，再決定資料能容許多大的過期程度。
需要熟悉的重要概念：
- TTL（存活時間）與過期機制
- Cache-aside（旁路快取）
- Write-through（同步寫入快取與資料庫）
- Write-back（先寫快取，之後再回寫資料庫）
面試時應該思考：**瓶頸在哪裡？資料最多可以容許多舊？**
## 3. CAP 定理
CAP 代表分散式系統中的一致性（Consistency）、可用性（Availability）與分區容錯性（Partition Tolerance）。
常見說法是「三者只能選兩個」，但影片指出這個說法並不完整，因為在真實的分散式系統中，網路分區 (P) 無法完全避免。
重點：
- 分區容錯性 (P) 不是可自由選擇的功能，而是分散式系統**必須**面對的現實。
- 發生網路分區時，實際上的選擇通常是在一致性與可用性之間取捨。
- \*一致性：\*\*每次讀取都能取得最近一次成功寫入的資料。
- \*可用性：\*\*系統一定會回應請求，但回傳的資料不一定是最新版本。
- 不需要為整套系統做出單一、全域性的選擇。
- 同一個系統中的不同功能，可以採用不同的一致性策略。
	- 金融交易、權限與存取控制通常需要強一致性。
	- 動態消息或內容 Feed 通常可以接受最終一致性（Eventual Consistency）。
面試時不要只說「我會使用最終一致性」，而應該說明：**哪些操作需要哪一種保證，以及原因是什麼。**
## 4. 訊息佇列（Message Queues）
訊息佇列可以將服務彼此解耦，避免一個服務必須同步等待所有下游操作完成。
重點：
- 在同步訂單流程中，應用程式可能需要依序呼叫庫存、付款和通知服務，全部完成後才能回應使用者。
- 這會形成一連串相依關係：只要其中一項服務變慢或暫時故障，整個訂單流程就可能失敗。
- 使用 Kafka、Amazon SQS 或其他訊息佇列時，系統可以發布 `OrderPlaced` 之類的事件。
- 庫存、付款與通知服務可以各自獨立取得並處理事件。
- 如果通知服務暫時故障，訊息可以留在佇列中，等服務恢復後再處理。
- 這能提升系統韌性，並降低服務之間的直接耦合。
需要注意的是，影片中的訂單案例經過簡化。真實系統中的付款與庫存通常還要處理重試、冪等性、訊息順序，以及失敗後的補償機制。
面試時應該思考：**哪些操作必須立即完成？哪些操作可以非同步處理？**
## 5. 資料庫：SQL、NoSQL 與 ACID
SQL 與 NoSQL 的選擇，不是「舊技術對新技術」或「小型系統對可擴展系統」這麼簡單。真正的問題是應用程式需要哪些資料保證與存取模式。
### ACID 四項特性
- \*原子性（Atomicity）：\*\*交易必須全部成功或全部失敗。例如轉帳時，不能只從一個帳戶扣款，卻沒有把錢存入另一個帳戶。
- \*一致性（Consistency）：\*\*每次交易都必須維持資料庫規則、限制條件與有效狀態。
- \*隔離性（Isolation）：\*\*同時執行的交易不應互相干擾，導致無法預測的結果。
- \*持久性（Durability）：\*\*交易一旦提交，即使伺服器隨後故障，資料也不會消失。
### SQL 與 NoSQL 的取捨
選擇重點在於是否需要 ACID
- 當交易正確性與關聯式限制非常重要時，通常適合使用 SQL。
- NoSQL 經常放寬 Schema 或一致性要求，以換取彈性與更容易的水平擴展。
- 金融交易、庫存，以及任何無法接受部分寫入的操作，通常適合 SQL。
- 使用者活動動態、商品目錄與大規模即時分析，如果可以容許短暫的資料過期，可能適合 NoSQL。
- 真實的正式環境經常同時使用 SQL 與 NoSQL，讓不同資料庫負責最適合自己的工作負載。
面試時應該思考：**這個工作負載是否需要 ACID 交易？如果資料暫時不一致，會造成什麼後果？**
## 6. API 設計
API 是服務與所有客戶端之間的契約。API 發布後，修改它不只是更新內部程式碼，而是一次可能影響所有下游使用者的協調式遷移。
重點：
- REST 相對簡單、容易快取，也容易理解與文件化。
- REST 適合公開 API，以及需要穩定介面的團隊和客戶端。
- GraphQL 允許客戶端精確指定需要的欄位。
- 如果網頁、行動裝置及其他客戶端有不同的資料需求，GraphQL 可能很有幫助。
- REST 和 GraphQL 沒有絕對的優劣，兩者針對的是不同需求。
- 應該從一開始就規劃 API 版本控制。
- Endpoint 應該圍繞「資源」設計，而不是直接暴露內部操作方式。
- 應事先定義並記錄 API 契約，避免使用者只能自行逆向理解。
- API 變更必須考慮向下相容性及對下游系統的影響。
面試時應該思考：**我們提供給客戶端的是什麼契約？如何在不破壞既有客戶端的情況下演進？**
## 影片的最終建議
創作者建議一次學習一項原則，並把它套用到日常使用的真實產品，而不是只背定義。
例如：
- Spotify 可能在哪些不同層級使用快取？
- Venmo 的哪些操作需要強一致性？
- 哪些操作可以接受最終一致性？
- 哪些狀態應該存在應用伺服器，哪些應該放在共享儲存系統？
- 一個工作流程中的哪些步驟應該同步執行，哪些可以改成非同步？
學習系統設計時，最重要的習慣是反覆詢問：
1. **這個架構模式為什麼存在？**
2. **它解決了什麼問題？**
3. **它帶來了哪些代價與取捨？**
4. **目前這個系統的哪些部分真的需要它？**
