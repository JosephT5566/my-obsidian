# 目的
想要實現 Line 的 scheduled message
# 什麼是 webhook?
相較於從 client 端每次都透過 api 去被動尋問 service。Webhook 是在 service 段先註冊 url，每當事件觸發時，service 就會主動通知 url
以 line messaging-api 為例，可以在後台的 webhook 登記 url，提供用戶串接自己的機器人，官網文件有列出 received message object
[Receive messages (webhook)](https://developers.line.biz/en/docs/messaging-api/receiving-messages/#webhook-event-in-one-on-one-talk-or-group-chat)
但這開發功能僅限『官方帳號』使用，方辦實名管理以及後續收費
[建立 LINE 官方帳號 - LINE 官方帳號教學 \| STEAM 教育學習網](https://steam.oxxostudio.tw/category/python/example/line-biz.html#a3)

## Apple 捷徑搭配 n8n 實現自動化回覆
1. 部署 n8n service
2. 捷徑呼叫 n8n api 來使用 Line api，取得 chat id
3. 捷徑設定指定時間以及訊息內容，並送到 n8n
4. 指定時間到，n8n api 就會根據 chat id 跟訊息內容，利用 Line api 來發送訊息
5. 同時我想要知道有哪些訊息正在排程，以及可以編輯/刪除
issues:
1. 我們無法手動取得 chat id（這部分受到 line 的加密保護），然後透過 api 傳給 n8n。只能在透過 webhook 收到事件時，取得對應的 chat id
## 聊天過程都通過 n8n，利用指令觸發排程 function
1. 使用者發訊息 → LINE → n8n webhook
2. n8n 流程判斷是否為排程指令
3. 如果是排程指令 → 儲存至資料庫
4. 如果不是排程指令 → 照原樣回覆
5. n8n 排程執行流程（另一個 workflow）
issues:
1. Line message 發送與 n8n workflow 處理，在使用 hugging face 這類 service 時，每次的觸發都會算入一次資源使用，LINE webhook 是「即時請求」，會喚醒伺服器
	1. 這個流程會耗費 CPU、記憶體與啟動時間，在高頻率或多人使用時會導致：延遲變高（回覆慢）、免費配額很快用完（每次觸發都算一次 request）
## 進入 n8n 前，多增加一層 webhook 判斷
1. 自己架一層輕量 webhook 判斷層
	1. （推薦用 Cloudflare Workers 或 Vercel）
2. n8n 僅負責處理指令類訊息
3. 不跑指令的訊息，就直接 reply 或 log
